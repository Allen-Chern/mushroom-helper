# 蘑菇重生計時器 — 設計文件

## 1. 理解摘要

- **目的**:協助朋友圈(個位數~十幾人)協作記錄皮克敏手遊(Pikmin Bloom 類 AR 蘑菇試煉)中蘑菇的重新生長時間
- **核心流程**:建立「觀察集」→ 產生可分享連結 → 任何拿到連結的人都能批次上傳截圖 → 前端 OCR 辨識地標名稱與剩餘時間文字 → 使用者確認/修正 OCR 結果 → 依「照片時間 + OCR 剩餘時間 + 固定 5 分鐘緩衝」算出重生時間 → 存入觀察集共用清單
- **協作模型**:完全匿名、無帳號登入,連結即權限,任何人可自由新增/編輯/刪除任何項目(類似 Uber 團購的信任模式)
- **通知**:裝置/瀏覽器層級設定(存 localStorage),預設重生前 20 秒響鈴,只要分頁開著(前景或背景)即可觸發,不需支援分頁關閉/App 完全退出時的推播
- **排序**:倒數中的項目在前(越快重生越前面),已到期項目排到最後標示「已重生/待確認」,保留至有新照片更新
- **非目標**:不需要帳號系統、不需要主頁歷史清單(使用者自行保存/分享連結)、不儲存原始照片、緩衝時間固定不可調、不支援 App 完全關閉時的推播通知

## 2. 假設

1. 後端方案採 Firebase(BaaS),免費額度足夠這個規模,不需自建/維運伺服器
2. 倒數時間文字解析需涵蓋「N小時N分N秒」「N分N秒」「N秒」等缺省組合
3. 通知使用瀏覽器 Notification API + 音效,需使用者授權通知權限;拒絕的話僅在分頁前景時響鈴;不支援 Notification API 的瀏覽器自動降級為僅音效
4. 不需要任何圖片儲存服務,照片確認後即從記憶體捨棄,從未上傳雲端
5. 地標名稱的 OCR 自動猜測準確度偏低,預期使用者經常需要手動修正

## 4. 最終設計

### 4.1 整體架構

```
使用者瀏覽器 (手機/桌機)
  ├─ Vue 3 SPA (Vite 建構,部署於 Firebase Hosting)
  │    ├─ 首頁:建立新觀察集
  │    ├─ 觀察集頁 /set/:setId
  │    │    ├─ 即時清單 (Firestore onSnapshot 訂閱)
  │    │    ├─ 批次上傳 + OCR 確認流程 (Tesseract.js,瀏覽器內執行)
  │    │    └─ 通知設定面板 (存 localStorage)
  │    └─ 前端計時邏輯:每秒 tick 重新計算剩餘時間、排序、觸發到期通知
  └─ Firebase Firestore (唯一雲端依賴,無自建後端伺服器/API)
```

- 觀察集用一組短的隨機 ID 當作路由與 Firestore 文件 ID,分享連結格式類似 `https://xxx.web.app/set/AbC123`,連結本身就是存取憑證(無需登入)
- Tesseract.js 的中文語言包(`chi_tra`)延遲載入,第一次上傳照片時才下載,避免拖慢首頁載入
- PWA 只做到「可加到主畫面、離線瀏覽已快取頁面」,不做背景推播

### 4.2 資料模型(Firestore)

```
observationSets (collection)
  └─ {setId} (document)
       ├─ createdAt: timestamp
       └─ items (sub-collection)
            └─ {itemId} (document)
                 ├─ locationName: string        // 地標名稱,OCR 猜測或人工修正後的值
                 ├─ respawnAt: timestamp         // 計算出的重生時間點
                 ├─ photoTimeSource: "exif" | "lastModified" | "manual"
                 ├─ status: "counting" | "awaiting_confirmation"
                 ├─ updatedAt: timestamp
                 └─ updatedBy: string  // 裝置隨機 clientId,僅用於顯示"最後由誰更新",非帳號
```

**安全性提醒**:Firestore Security Rules 採「知道 setId 即可讀寫該文件底下所有資料」— setId 本身即為唯一存取控制,符合「連結即權限、任何人可自由編輯」的協作模型,不會有更細的權限管控。

### 4.3 核心使用者流程

**建立觀察集**:首頁點「建立新觀察集」→ 產生隨機 `setId` → 寫入 Firestore → 導向 `/set/{setId}` → 提示複製連結分享。

**批次上傳與 OCR 確認**:
1. 觀察集頁點「上傳照片」,可一次選多張
2. 逐張(非同時,避免手機負載過重)跑 Tesseract.js OCR
3. 每張處理完立刻顯示確認卡片:原圖縮圖、地標名稱猜測(取畫面上方 1/3 區域最長文字,可編輯)、剩餘時間猜測(用正規表達式比對「剩下(?:X小時)?(?:Y分)?(?:Z秒)?」格式,可編輯)、時間來源標籤
4. 確認/修正後計算 `respawnAt = 照片時間(lastModified 優先 EXIF,都無則強制手動輸入) + 剩餘時間 + 5 分鐘`
5. 若 `locationName` 與現有 `counting` 狀態項目重複,提示「更新既有項目」或「新增一筆」
6. 寫入 Firestore,原圖從記憶體捨棄
7. 全部跑完顯示「已新增 N 筆」摘要

**即時清單顯示**:所有開著該觀察集頁面的使用者透過 `onSnapshot` 即時收到新增/修改/刪除,清單自動重新排序與重繪。

### 4.4 倒數計時 / 排序 / 通知邏輯

- 全域 `setInterval` 每秒觸發,重新計算所有 `counting` 項目的 `remainingMs`
- 排序:`counting` 依 `remainingMs` 由小到大排最前;`awaiting_confirmation` 排最後,依 `respawnAt` 新到舊排
- 到期轉換:`remainingMs <= 0` 時前端把該文件 `status` 更新為 `awaiting_confirmation`(多人同時寫入同樣的值無害)
- 通知觸發:每次 tick 檢查 `remainingMs` 是否「跨過」使用者設定的提醒秒數(預設 20 秒),觸發後在前端記憶體標記「已通知過」(不同步到 Firestore),播放音效 + 若已授權則同時發 Notification
- 每個裝置依自己的 localStorage 設定獨立判斷是否要響,同一份倒數資料不同人可設定不同提醒時間點

### 4.5 錯誤處理與邊界情況

- Tesseract.js 載入失敗:顯示錯誤 + 「重試」按鈕,確認表單仍可手動填寫
- OCR 完全辨識不到時間格式:欄位留白標紅提示,必填欄位為空時「加入」按鈕停用
- Firestore 讀取失敗:顯示連線提示,搭配 SDK 內建離線快取自動同步
- Firestore 寫入失敗(離線):暫存本機,顯示「等待網路恢復後送出」,不阻塞後續照片處理
- 重複地點比對時機:使用者按下「加入」的當下即時比對(涵蓋同批次內重複的情況)
- 計算出的 `respawnAt` 明顯不合理(早於現在超過閾值):顯示警告但不強制阻擋送出
- 瀏覽器不支援 Notification API:自動降級為僅音效通知

### 4.6 測試策略

- 手動測試:核心流程(建立觀察集、上傳確認、多分頁即時同步、通知觸發)用 2 台裝置手動走過
- 單元測試(Vitest):倒數時間文字解析函式、重生時間計算函式、排序函式 — 三個純函式,邏輯複雜且容易改壞,值得覆蓋
